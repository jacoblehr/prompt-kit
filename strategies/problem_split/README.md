# Problem Split

Use when:

- the task feels vague, tangled, or overloaded
- framing is the bottleneck

Helps prevent:

- tackling the wrong subproblem
- mixing framing, decision, and execution together
- hidden dependencies staying invisible

How to use it:

1. restate the problem
2. identify distinct subproblems
3. separate knowns from unknowns
4. identify which subproblem should be handled first

Quick invocation:

```text
Apply the PROBLEM SPLIT strategy.
Break this into distinct subproblems, surface open questions, and suggest the best order of attack.
Keep the decomposition practical rather than exhaustive.
```
